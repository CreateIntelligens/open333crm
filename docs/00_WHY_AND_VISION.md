# 00 — 為什麼重建？舊系統痛點 + 新系統願景

## 現有系統（舊版 LINE OA 後台）的致命問題

### P1 架構問題

| 問題 | 影響 |
|------|------|
| 沒有前後分離，PHP 直出 HTML | 無法給第三方 App / 行動端使用；改版成本極高 |
| 無對外 API | 客戶無法自行整合；無法賣 API 服務 |
| 渠道與邏輯嚴重耦合 | 所有流程、事件都預設是 LINE；加入 FB/WhatsApp 幾乎要重寫 |
| 功能疊床架屋、無 Plugin | 每個需求直接寫進核心，累積技術債 |

### P2 CRM 能力不足

| 缺失功能 | 真正 CRM 應有的 |
|----------|----------------|
| 無 Case 開/處理/關閉 | 客服問題可追蹤、可量化、有 SLA |
| 無聯繫人資料更新機制 | 客戶資訊永遠停在第一次綁定 |
| 無 1on1 事件升級 | 機器人無法判斷何時升人工，或升給誰 |
| 對外聲稱 CRM，實為群發工具 | 不符客戶期待，增加客訴風險 |

### P3 成長天花板

| 問題 | 影響 |
|------|------|
- 無法橫向擴充渠道：加入新 IM 成本不可接受
- 無預約/報名等垂直場景模組：競品差異化不足
- 無關係鏈：無法建立「用戶 A 推薦用戶 B」的裂變資料
- 無粉絲互動工具：缺乏投票、競猜、表單等行銷互動，無法沉澱粉絲
- 無行為追蹤：無法知道「誰」點了「哪個連結」，行銷效果無法歸因

## 新系統願景（open333CRM Lite）

### 核心定位

> **一個中小企業從「第一次觸及顧客」到「問題解決」再到「持續經營」的整合操作界面**，
> 覆蓋多渠道訊息、聯繫人管理、案件處理、自動化行銷、粉絲互動五大支柱。

### 設計哲學

1. **渠道無關（Channel Agnostic）**
   所有 IM 渠道（LINE / FB / WebChat）皆以統一抽象層接入，業務邏輯完全不知道底層是哪條渠道。

2. **事件驅動（Event Driven）**
   用戶行為、案件狀態、自動化規則，全部以事件流串聯，方便擴充和審計。

3. **插件優先（Plugin First）**
   渠道是插件、AI 是插件、行銷模組是插件。核心保持最小化、穩定化。

4. **真 CRM 能力**
   Case 生命週期管理 + 聯繫人基礎資料 + 指派機制 + SLA 追蹤。

5. **Lite 不代表功能少**
   Lite 是指部署簡單（單企業部署），不是指功能砍半。MVP 後可平滑升級到 SaaS 多租戶。

6. **開放平台（Open API）**
   企業的 ERP、電商、IoT 系統可透過 API 將資料推入 CRM，使 CRM 成為所有客戶互動的中心樞紐。API 呼叫量也是可以計費的商業模式。

7. **行為可追蹤（Trackable）**
   所有對外連結都包裝為短連結（結合 LIFF），每次點擊都能識別身份、記錄行為、自動貼標，讓行銷效果可歸因。

### 不只是 CRM — 三層產品價值

```
第一層：客服中心
  → 統一收件匣、Case 管理、SLA、客服績效

第二層：行銷引擎
  → 廣播、自動化規則、分群、Flex 模板、短連結追蹤

第三層：粉絲經營平台
  → Fan Portal（投票/表單/競猜）、積分系統、會員中心
  → 透過 LINE LIFF 無縫導流，一鍵進入，身份自動帶入
```

### 目標客戶（第一個客戶）

- **老牌家電業者**
- 需整合：LINE OA + FB Messenger + Web Chat
- 主要場景：售後客服、產品諮詢、維修預約
- 行銷需求：保固提醒廣播、活動投票、新品問卷
